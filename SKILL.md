# 工作日报（日报）

## 何时使用

- 用户要写/改 **日报**、**工作日报**、**今日总结**、**工作内容**。
- 用户提供零散要点，需要整理成带序号、每条含「短标题 + 说明段落」的长文。
- 用户希望结合 **本对话里问过的问题、讨论过的方案** 和 **项目里写过的代码** 一起归纳成日报。

## 素材来源（可与 IDE 内 AI 助手对话 / 代码结合）

写日报时 **优先** 从以下来源提炼「工作内容」，下条读文件或 `git` 前可先基于已有对话与附件：

1. **当前对话**（含 VS Code、Cursor、Windsurf、Codex 等任意 Agent/助手会话）：用户提过的问题、选型讨论、结论、待办；用 **业务语言** 概括，不要逐句复述聊天。
2. **代码与仓库**（用户未禁用时）：
   - 用户 **@ 文件/文件夹** 或明确路径 → 可阅读相关文件，从 **模块职责、关键改动意图** 概括进日报（类名/文件名可择要出现，不必贴代码）。
   - 用户说「按今天改动」「结合仓库」→ 可用 `git diff` / `git log`（今日范围）或最近编辑文件辅助归纳；若仓库与对话无关，以对话为准并 **不强行编造** 与代码的关联。
3. **诚实边界**：对话或 diff 里 **没有** 的信息不写成已完成的成果；推断处用「推进了」「梳理了」等与实际证据力度一致的说法。实验性、未合并的改动写清楚阶段（如「验证中」「分支上」）。

### 今日有代码产出时（与代码相关的条目必写清）

若当天确实存在 **编写或修改的代码**（对话产出、本地提交、未提交改动均可，以可核对为准），凡日报条目 **对应这段代码**，正文里须写清楚下面三类信息（可融成一段，不必分三小节）：

1. **改动情况**：改了哪些模块/文件或哪些能力（新增、重构、修 bug、调参）；与之前行为的差异或范围。
2. **技术使用**：用到的语言/框架/库/模式（如 React hooks、某 API、状态管理、构建方式等），只写与理解该条工作相关的要点，不堆砌无关栈信息。
3. **代码作用**：这段实现 **解决什么问题**、**对外行为** 是什么（输入输出、边界、与上下游关系），避免只写「写了某某文件」而无功能说明。

与代码无关的条目（纯会议、纯调研、纯文档）仍按常规叙述，不强行塞代码细节。无代码产出日忽略本节。

## 版式（默认）

1. **日期**：单独一行，格式 **`YYYY年M月D日`**（月份、日期 **不补零**，如 `2026年4月1日`）。用户未指定日期时，使用 **当天**（按用户语境中的「今天」）。
2. **标题行**：空一行后写 **`工作内容：`**，再空一行。
3. **条目**：`1.` `2.` … 每条 **先一行短标题**（概括该条），**换行后**写一段或数段说明（背景、做了什么、结论或取舍）。风格参考 [examples.md](examples.md)。
4. **条数**：用户 **未说明写几条** 时，默认 **`5` 条**。若用户明确说「写 3 条」「两条即可」等，按用户条数。
5. **要点不足 5 条时**：在 **不编造事实** 的前提下，可将用户给出的大块工作 **合理拆成多条**（按阶段、按子任务、按问题—方案—结果）；若仍不足且无法诚实拆分，输出实际条数并 **简短提示** 用户可补充哪些主题，勿用虚构项目凑满 5 条。

## 写作原则

- 用 **结果与判断** 写清楚「做了什么、为什么、得出什么结论」，避免空洞套话；**有代码时** 相关条目同时满足上文「改动情况 / 技术使用 / 代码作用」。
- 敏感信息（未授权的客户名、保密数据）用泛化表述。
- 不编造数据与未发生的工作；用户未给的数字不写或标「待补充」。

## 安装位置（各编辑器）

本 skill 需放在 **你所用工具的技能目录** 下才会被加载；仓库里的副本仅作版本管理，请按需同步到本地技能路径，例如：

| 环境 | 常见路径（将本仓库的 `daily-report` 文件夹整份放入或符号链接） |
|------|----------------------------------------------------------------|
| Cursor | `~/.cursor/skills/daily-report` |
| OpenAI Codex CLI / 使用 `~/.codex/skills` 的工具 | `~/.codex/skills/daily-report` |
| 其他 Agent 编辑器 | 以该工具的文档为准（多为用户目录下的 `skills` 子目录） |

VS Code 若通过扩展或配置挂载技能目录，把同名文件夹放到扩展要求的 `skills` 路径即可；核心仍是 **一份 `SKILL.md` + 同目录资源**，与具体品牌无关。
