# 日报模板与版式说明

## 默认版式（纯文本 / Markdown 均可）

```text
YYYY年M月D日

工作内容：

1. 本条短标题（概括主题）  
   说明段落：背景或现状、具体动作、结论或取舍。可多句。

2. 本条短标题  
   说明段落……

3. …
4. …
5. …
```

### 约定

| 项目 | 说明 |
|------|------|
| 日期 | 单行；`M`、`D` **不补零**（例：`2026年3月31日`、`2026年4月1日`）。未指定时为 **当天**。 |
| 工作内容 | 固定用 **`工作内容：`** 四字，后空一行再写序号。 |
| 每条结构 | **`序号. 标题`** 一行；换行后写正文；标题宜 **一行内说完**，正文可稍长。 |
| 条数 | 用户未说写几条时，默认 **5 条**。 |

## Markdown 中的换行

每条「标题」与「正文」之间用 **行末两个空格** 再换行，或用一个空行把长正文包在列表项内（与编辑器预览习惯一致即可），保持阅读上与用户示例一致：**标题一行、下面跟段落**。

## 与用户指定条数冲突时

用户说「只写两条」→ 输出 2 条。用户未说条数 → **5 条**。

## 与 Cursor 对话、代码结合时

- **对话**：从用户问过什么、定了什么方案里抽条目标题与说明，写成工作叙述，而非聊天记录摘要。
- **代码**：必要时结合用户 @ 的路径或今日 `git` 改动；避免大段贴代码；内部类名、文件名可作佐证，敏感仓库名可模糊化。

### 当天有代码产出时，与代码相关的条目须覆盖

| 维度 | 写什么 |
|------|--------|
| **改动情况** | 动到哪些文件/模块、新增还是修改、行为与之前有何不同 |
| **技术使用** | 关键语言/框架/库/模式（与理解该条工作相关的部分即可） |
| **代码作用** | 解决什么问题、对外可观察的行为、与上下游的关系 |

纯非编码工作条目不要求强行写代码三要素。
